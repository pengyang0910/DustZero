# app_server 具体代码逻辑

> 本文档描述 app_server（即 appTask 进程，云端/App 代理）的内部代码逻辑，包括启动流程、多线程模型、Tuya SDK 对接、命令转换、RPC 交互、数据上报、LCM 订阅、固件升级、WiFi 配网与故障管理，不含源码。

---

## 1. 启动流程

### 1.1 创建实例
在堆上构造应用服务器核心对象，该对象封装了 LCM 通信、Tuya SDK 交互、RPC 客户端、数据缓存与压缩等全部能力。

### 1.2 调用 Start 方法
进入正式的启动序列。

### 1.3 绑定 CPU 核心
将主线程绑定到预定义的 APP 专用 CPU 核心上，避免与其他高负载进程争抢。

### 1.4 设置线程名
将主循环线程命名，便于系统调试与监控。

### 1.5 进入守护循环
主线程以每秒一次的频率休眠，同时检测 WiFi 重置标志。若检测到重置完成状态，则调用重新初始化方法，重新初始化 Tuya SDK 线程和地图传输线程。

---

## 2. 多线程模型

AppServer_t 内部维护三个工作线程，职责划分清晰：

### 2.1 appTask 线程（主业务线程）
- **职责**：作为核心调度线程，以约 5 毫秒为周期循环执行。
- **工作内容包括**：
  - 轮询检测硬件按键（WiFi 重置组合键）。
  - 从 Tuya SDK 获取 DP 命令（包括 Object DP 和 Raw DP）。
  - 通过 RPC 客户端将命令转发给 hmi_server。
  - 驱动 LCM 消息处理。
  - 触发地图与轨迹的压缩与缓存（每 40 个 tick，约 200 毫秒）。
  - 上报错误码与基站状态（每 10 个 tick，约 2 秒）。
  - 上报设备静态信息（首次运行时上报一次产品型号、固件版本、SN 号等）。

### 2.2 TuyaSdk 线程
- **职责**：独立运行涂鸦 IPC SDK 的主循环。
- **生命周期**：由启动函数启动，内部完成 SDK 初始化、MQTT 上线等待、P2P 初始化、本地存储初始化等，随后进入事件循环直到进程退出或收到重置信号。
- **数据交互**：SDK 通过全局互斥锁保护的指针和布尔标志，将接收到的云端 DP 命令投递给 appTask 线程消费。

### 2.3 sweepDetect 线程（地图/轨迹传输线程）
- **职责**：专门处理扫地机地图相册数据的 P2P 传输。
- **工作方式**：循环扫描会话控制块数组，当检测到 App 端发起地图或轨迹下载请求时，从指定的临时文件路径读取压缩后的地图/轨迹二进制流，通过 SDK 接口分段发送给 App。支持流传输和文件传输两种模式。

## 3. Tuya SDK 对接逻辑

### 3.1 长连接建立流程
Tuya SDK 的启动由独立的 TuyaSdk 线程完成，流程如下：

1. **读取激活凭证**：从配置文件中读取设备的 UUID、AuthKey、版本号，以及日志配置，将其注入 SDK 环境变量。
2. **配置 SDK 运行参数**：
   - 设置产品 ID、UUID、AuthKey、软件版本号。
   - 配置设备类型为低功耗设备。
   - 配置媒体流参数（主/子码流分辨率、帧率、编码格式、音频采样率等），尽管 app_server 不传输真实视频，但需满足 SDK 初始化要求。
   - 配置本地存储、云存储、P2P、AI 检测、门铃等模块的使能状态。
   - 注册网络状态变化回调、DP 命令回调、Raw DP 命令回调、查询回调、设备重置/重启回调、OTA 升级通知回调。
3. **初始化 RingBuffer**：为音视频通道创建内部环形缓冲区。
4. **启动 SDK**：调用 SDK 启动函数，内部完成 MQTT 连接、CoAP 服务、P2P 初始化等。
5. **等待 MQTT 上线**：线程阻塞循环，直到网络状态回调标记 MQTT 已上线。
6. **初始化扫地机专用 SDK**：调用扫地机 SDK 初始化函数，注册扫地机相册传输回调。

### 3.2 DP 命令的接收和解析流程
云端下发的 DP 命令分为两类：

- **Object DP（结构化数据）**：由 SDK 的结构化命令回调接收。回调内部先将命令缓存到全局指针，设置命令到达标志通知 appTask 线程；随后阻塞等待最多 5 秒，直到 appTask 处理完毕并清除标志，最后向云端回执处理结果。
- **Raw DP（二进制数据）**：由 SDK 的原始命令回调接收。处理方式与 Object DP 类似，但缓存的是原始字节流。对于特定命令（如多地图保存、多地图删除、勿扰时间设置等），回调内部直接调用 SDK 接口完成处理并回执，不再转发给 appTask。

appTask 线程中的命令消费函数负责处理 Object DP：
- 判断命令类型（布尔、数值、字符串、枚举）。
- 将 DP ID 和值转换为内部结构体。
- 对于部分特定 DP（如重置地图、耗材寿命查询等），直接在 appTask 中完成云端回执，不再向下转发。

### 3.3 常见 DP 命令的处理

| DP 类型 | 典型命令 | 处理方式 |
|---|---|---|
| 布尔型 | 开关机、回充、暂停、继续、儿童锁、屏幕灯、自动集尘、自动烘干、地毯增压、防打扰 | 转换为 RPC 命令发给 hmi_server，同时更新本地状态并回执 |
| 枚举型 | 清扫模式（扫/拖/扫拖）、吸力档位、水箱档位、地毯清洁模式、工作模式 | 同上，其中部分档位值需做范围钳制和偏移映射 |
| 数值型 | 音量设置、集尘间隔、边刷/滚刷/滤网/拖布寿命 | 部分直接回执（如耗材寿命查询），部分转发 hmi_server |
| 字符串型 | 设备型号、产品名称、SN 号、固件版本 | 首次连接时主动上报 |
| Raw 型 | 虚拟墙、禁区、选区清扫、划区清扫、地图保存/删除、房间分割/合并 | SDK 层直接解析坐标并缓存；部分直接调用多地图接口 |

## 4. 本地命令转换

appTask 作为云端与本地机器人系统的翻译层，其核心转换逻辑如下：

### 4.1 云端 DP → RPC 命令
- 布尔型 DP（如开关机、回充）被封装为结构化命令，通过 RPC 调用发送给 hmi_server。
- Raw DP 被封装为原始命令结构，通过 RPC 调用发送，用于虚拟墙、禁区、选区等复杂地理信息的传递。

### 4.2 参数映射
- **吸力档位**：云端枚举值与本地风机等级做直接映射，并做越界保护。
- **水箱档位**：云端枚举值映射到本地水箱等级，做越界保护。
- **清扫模式**：云端枚举值与本地枚举值做交叉映射。
- **音量**：本地值范围映射为云端百分比。
- **耗材寿命**：本地以分钟/次为单位，云端直接显示该数值。

### 4.3 状态同步的副作用
- 当收到"开始清扫"命令时，appTask 会主动向云端批量上报多个关联 DP（如取消暂停、取消回充、设置状态为清扫中），确保 App 端状态一致性。
- 同理，"回充"命令会触发停止清扫、取消洗拖布等状态的批量上报。

## 5. 与 hmi_server 的 RPC 交互

### 5.1 连接流程
appTask 线程在启动时创建 RPC 客户端实例，尝试连接本机的 hmi_server RPC 服务端口。连接参数包括：
- 等待连接超时 20 秒。
- 启用自动重连。
- 启用自动心跳。

### 5.2 调用的接口和参数构造
appTask 主要调用两个 RPC 接口：
- **App 命令接口**：参数为结构化命令，包含 DP ID、类型、字符串形式的值、时间戳。hmi_server 收到后将其加入 App 命令队列，供后续状态机消费。
- **原始命令接口**：参数为原始命令结构，包含 DP ID、数据长度、原始字节数组、时间戳。用于虚拟墙、禁区、选区等复杂地理信息的传递。

### 5.3 连接断开与重连逻辑
- RPC 客户端配置了自动重连与自动心跳，底层库负责在连接断开时持续尝试恢复。
- appTask 在每次发送前检查连接状态，若未连接则跳过本次命令转发，避免阻塞。
- 存在一个特殊的初始化握手逻辑：连接状态标志从 1→2→3 变迁。当首次检测到 hmi_server 的 LCM 广播时置为 2，随后通过 RPC 发送一个特殊的连接确认 DP 给 hmi_server，成功后置为 3，标志 appTask 与 hmi_server 的通道正式就绪。

## 6. 状态与数据上报

### 6.1 机器人状态转换为 Tuya 状态并上报
appTask 订阅了 hmi_server 广播的机器人信息通道，在回调中完成状态转换与上报：
- **基站状态**：直接映射基站状态字段。
- **机器人工作状态**：直接映射机器人状态字段。
- **电池电量、清扫面积、清扫时间**：直接映射对应字段。
- **儿童锁、屏幕灯**：映射布尔字段。
- **变化触发上报**：只有字段值发生变化时才进行异步上报，减少网络流量。

### 6.2 地图数据的接收、压缩、分包上报流程
1. **接收**：通过 LCM 订阅地图通道，在回调中接收栅格地图消息，缓存地图的宽度、高度、分辨率、原点坐标、房间属性数组、栅格数据。
2. **坐标系转换**：将地图原点坐标和充电桩坐标从浮点米转换为 SDK 需要的整数格式，并做左右镜像翻转。
3. **房间数据拼接**：在栅格数据末尾追加房间信息，包括房间 ID、清扫顺序、清扫次数、拖地次数、颜色、禁止扫/拖标志、吸力、水量、拖地标志、房间名称、以及房间顶点坐标。
4. **LZ4 压缩**：使用 LZ4 算法对整个地图数据包进行压缩。
5. **写入流文件**：构造二进制文件头，写入临时流文件。该文件即为后续 P2P 传输给 App 的地图数据源。

### 6.3 轨迹数据的上报
1. **接收**：通过 LCM 订阅轨迹通道，在回调中接收轨迹消息。轨迹支持增量追加，当收到带清除标记的帧时，清空历史轨迹。
2. **坐标转换**：将每个轨迹点的浮点坐标转换为 16 位整数地图坐标，并根据轨迹类型在坐标最低位嵌入类型标记。
3. **LZ4 压缩**：对转换后的轨迹点数组进行 LZ4 压缩。
4. **写入流文件**：构造路径文件头，写入临时流文件。

### 6.4 清扫记录和统计信息的维护
当检测到清扫完成标志时，触发记录上传：
1. 读取当前的地图流文件和轨迹流文件。
2. 追加当前虚拟墙数据和禁区数据，拼接成一个完整的清扫记录二进制包。
3. 生成记录 ID。
4. 调用 SDK 接口将记录上传至涂鸦云端。
5. 同时维护本地累计统计文件，更新并上报累计清扫面积、累计清扫次数、累计清扫时间，以及边刷、滚刷、滤网、拖布、传感器、底板的剩余寿命（随清扫时间递减）。

## 7. LCM 消息订阅逻辑

appTask 进程通过 LCM 与系统内其他进程通信，订阅的通道如下：

| 通道名 | 用途 | 队列容量 |
|---|---|---|
| AppMap | 接收地图栅格与房间属性 | 5 |
| AppTraj | 接收清扫/导航轨迹 | 1 |
| hmiToApp 状态广播 | 接收 hmi_server 广播的机器人实时状态 | 无限制 |
| hmi_publish_para_app | 接收 hmi_server 广播的机器人参数 | 无限制 |
| slam_publish_para_app | 接收 slam 模块广播的房间分割/合并/禁区/虚拟墙等几何参数 | 无限制 |
| BlockInfoFromSlam | 接收房间区块的原始顶点坐标，用于地图数据拼接 | 4 |

**处理与缓存逻辑**：
- 地图和轨迹回调遵循"仅缓存、不处理"原则，将原始数据拷贝到成员变量后立刻返回，避免阻塞 LCM 回调线程。
- appTask 主循环中的压缩函数在独立时机（每 200ms）对缓存数据进行消费、压缩和写文件，实现生产-消费解耦。
- 机器人状态回调中直接进行 Tuya 状态转换和上报，但同样遵循"变化才上报"原则。
- 房间参数回调中，将浮点坐标转换为地图坐标系下的双字节数据，缓存到全局结构，供后续地图拼接或 Raw DP 回执使用。

## 8. 固件升级相关逻辑

涂鸦云端触发 OTA 时，SDK 通过注册的升级通知回调将固件信息下发：

1. **接收升级通知**：回调函数获取固件下载地址、目标版本号、文件大小。
2. **创建本地文件**：在预定路径创建文件。
3. **分片下载与写入**：SDK 自动分片下载，通过回调将数据写入本地文件。下载过程中，每收到一个分片即计算下载百分比，并按 50% 折算为上报进度（预留 50% 给刷机操作）。
4. **下载完成处理**：
   - 关闭文件句柄。
   - 播放语音提示"升级中"。
   - 将升级包重命名为目标路径。
   - 修改配置文件中的版本号字段为新版版本号。
   - 上报进度 99%。
   - 调用系统重启命令完成升级。

## 9. WiFi 配网与重置逻辑

### 9.1 WiFi 配网
Tuya SDK 通过适配层与系统 WiFi 交互：
- **Station 连接**：收到配网信息后，调用系统命令连接路由器，成功后保存配置。
- **AP 扫描**：调用扫描命令扫描周边 AP，解析 BSSID、信道、RSSI、SSID 返回给 SDK。
- **SoftAP 模式**：配网时启动开放热点，供 App 连接并传输 WiFi 信息。
- **Sniffer 模式**：在 SmartConfig 配网时，创建原始套接字线程捕获空口数据包，解析其中的配网信息。

### 9.2 WiFi 重置逻辑
appTask 主循环持续检测硬件按键组合（Fn 键 + Power 键）：
- 通过 sysfs 读取两个按键的电平状态。
- 若两者同时按下超过一定时间，则触发 WiFi 重置。
- 重置操作调用 SDK 的网关重置函数，清除配网信息和激活状态。
- 主函数检测到重置标志后，将 WiFi 状态机恢复为初始态，并重新创建 TuyaSdk 线程和 sweepDetect 线程，实现"重置后自动进入配网模式"。
