# navclean 具体代码逻辑

> 本文档描述 navclean（主导航清扫进程）的内部代码逻辑，包括启动流程、调度循环、任务状态机、插件加载、异常处理与通信交互，不含源码。

---

## 1. 启动流程

navclean 进程的启动遵循严格的五级顺序：

### 1.1 信号注册
进程首先注册中断与终止信号的处理器，用于捕获外部终止指令并设置全局原子标志位，通知主循环优雅退出。

### 1.2 调度器实例化
创建 `TaskScheduler_t` 对象，作为 navclean 的唯一顶层控制器。

### 1.3 调度器初始化
此阶段按固定顺序完成内部子系统构建：
- **NavBridge 初始化**：建立与底层硬件、SLAM、地图服务的 LCM 通信通道；启动独立的 LCM 订阅线程循环接收消息；同时创建三个内置规划器实例（弓字路径规划器、纯跟踪规划器、DStar 内部规划器）。
- **TaskPools 初始化**：将 NavBridge 指针注入任务池，随后实例化所有内置导航任务（共 18 个），包括待机、弓字清扫、沿墙、房间清扫、旋转、遥控等，全部存入 NavTask 字典。
- **任务列表加载**：将 NavTask 字典和 DaemonTask 字典中的任务指针分别复制到调度器的两个栈容器中。
- **RPC 初始化**：将 NavBridge 指针注入 RPC 模块，为后续状态上报做准备。
- **初始任务启动**：从任务池中取出 IdleTask（待机任务），设为当前任务并调用其 Start 方法，使其进入 RUNNING 状态。

### 1.4 动态插件加载
确定插件目录（优先读取环境变量，否则取可执行文件所在目录的相对路径）。通过全局辅助函数依次加载：
- **NavTask 插件**：回充任务、地图探索任务、重定位任务。
- **DaemonTask 插件**：异常处理任务、虚拟碰撞任务、基站守护任务。

每个插件通过动态库加载方式打开对应命名的 `.so` 文件，根据标志决定注册到 NavTask 池还是 DaemonTask 池。

### 1.5 调度器启动
打开 RPC 服务线程，然后启动 20ms 周期的核心调度线程。主线程进入阻塞循环，每 100ms 检测一次退出标志，直到收到终止信号后调用 Stop 完成优雅退出。

## 2. 20ms 主循环调度逻辑

调度线程是 navclean 的心跳，使用高精度时钟维持约 20ms 的固定周期，每周期在互斥锁保护下依次执行六大阶段：

### 2.1 cleanTaskUpdate（驱动当前主任务）
检查当前主任务指针是否有效，若有效则调用其 Update 方法。随后检测任务状态：
- 若任务已变为 STOP，则读取其预设的下一个任务名，调用任务切换逻辑进行切换。
- 若任务未指定后续任务，默认回退到 IdleTask。

### 2.2 deamonTaskUpdate（驱动守护任务）
遍历守护任务栈容器，对每个状态为 RUNNING 的守护任务调用其 Update。守护任务与主任务并行运行，不受主任务切换影响，常用于持续监控类逻辑。

### 2.3 robotStateUpdate（机器人状态监测）
从 NavBridge 聚合最新状态，填充机器人数据结构（电量、充电状态、位姿、错误码、清扫模式、当前任务名）。该阶段包含两层关键决策：
- **低电量保护**：若电量低于阈值且机器人正在执行非待机、非回充任务，则自动设置回充标志和异常类型，强制触发回充流程。
- **异常分类**：若 NavBridge 上报了硬件错误码，按位解析为碰撞、轮子卡住、悬崖检测等异常类型，并将机器人状态置为错误。

同时根据当前任务和充电状态更新全局状态机（待机 / 工作中 / 充电）。

### 2.4 DataExchangeWithRpc（RPC 数据同步）
将上一步聚合的机器人和状态机写入 RPC 模块的缓存区，供外部 HMI 查询。

### 2.5 hackHandle（调试与外部命令处理）
这是外部干预的集中处理枢纽，按优先级依次检查 NavBridge 中的标志位：
- **暂停**：若暂停标志为真且当前任务在运行，调用 Pause；若暂停标志解除且任务处于暂停，调用 Resume。
- **结束作业**：若作业完成标志为真，重置所有相关标志并将当前任务切回 IdleTask。
- **遥控模式**：若遥控模式标志为真，切换到遥控任务。
- **强制回充**：若回充标志或外部触发器为回充，停止当前主任务，启动回充任务。
- **开始清扫**：若清扫标志为真且当前为待机状态，切换到清扫准备流程。

### 2.6 keyBtnHandle（实体按键）
当前为预留接口，尚未实现具体逻辑。

## 3. 任务状态机与生命周期

所有任务均继承自统一的抽象基类，其生命周期状态机包含四个状态：STOP、RUNNING、PAUSE、READY（实际主要使用前三者）。

### 3.1 状态转移路径
- **启动**：STOP → RUNNING。调用 Start 时，先清空后续任务名和挂起标志，加载 Hook 函数栈，然后执行子类的初始化逻辑，接着一次性执行启动前 Hook 栈，最后将状态置为 RUNNING。
- **停止**：RUNNING 或 PAUSE → STOP。调用 Stop 时，执行子类清理逻辑，一次性执行停止前 Hook 栈，清空主 Hook 栈，状态回退为 STOP。
- **暂停**：RUNNING → PAUSE。设置挂起标志，执行挂起前逻辑。
- **恢复**：PAUSE → RUNNING。清除挂起标志，执行恢复前逻辑。

### 3.2 每帧更新逻辑
只有状态为 RUNNING 且未被挂起的任务才会执行更新。每 20ms 的 Update 严格按照以下顺序执行：
1. 遍历并执行主 Hook 栈中的所有 Hook 函数（这些 Hook 通常由父任务或插件注入，用于插入监控、异常拦截等横切逻辑）。
2. 执行子类的核心循环逻辑。

### 3.3 任务切换逻辑
执行任务切换时：
1. 通过任务池按名字查找下一个任务，若找不到则直接返回。
2. 若下一个任务与当前任务相同且当前任务未停止，则跳过。
3. 若当前任务仍在运行，先调用其 Stop 强制结束。
4. 保存当前任务为上一个任务指针，注入到新任务的反向链字段中，形成单向任务链。
5. 调用新任务的 Start，将其置为当前任务，同时更新 NavBridge 中的当前任务名字段。

## 4. NavBridge 资源聚合逻辑

NavBridge 是 navclean 进程与外部世界的唯一数据枢纽，采用聚合器加消息总线的架构。

### 4.1 初始化与通信
初始化阶段创建消息上下文对象，启动一个独立的 LCM 订阅线程。该线程以固定超时循环处理以下通道：
- **里程计**：底盘里程计位姿和 SLAM 定位位姿。优先使用 SLAM 位姿作为机器人当前位姿，若定位未运行则回退到里程计。
- **电源与碰撞**：电量百分比、充电状态、碰撞传感器左右状态。
- **地图与区块**：地图元数据、分辨率、原点、充电桩位置、房间/区块多边形列表及清扫状态。
- **状态与速度**：App 在线状态、定位质量分数、外部遥控速度指令。

### 4.2 坐标变换系统
NavBridge 内部维护变换矩阵，通过同时收到的里程计位姿和 AMCL 位姿实时计算 SLAM 坐标系到 Odom 坐标系的刚性变换。提供多组接口实现双向转换，供所有任务在导航和目标点下发时统一坐标系。

### 4.3 规划器与硬件访问
NavBridge 在初始化中直接创建三个规划器对象，任务通过指针访问器访问。同时提供硬件/服务指针（当前为前向声明占位，待对应库接入后可直接使用）。另外提供速度发布接口，将任务计算出的线速度、角速度通过速度指令通道下发给底盘。

### 4.4 地图区块管理
NavBridge 内部缓存从 SLAM 接收到的完整区块数组，提供查询下一个未清扫区块、按 ID 获取房间、标记区块已清扫、清空已清扫记录等线程安全操作。

## 5. 插件加载机制

任务池采用内置任务加动态插件的双层架构。

### 5.1 内置任务
初始化方法中批量实例化 18 个内置 NavTask，涵盖清扫流程所需的全部基础能力。这些任务在编译期链接，生命周期由 TaskPools 管理。

### 5.2 外部插件加载
插件加载实现标准 Unix 动态库加载流程：
1. 以延迟解析模式打开指定路径的 `.so`。
2. 查找导出符号工厂函数，其签名为接收 NavBridge 指针和字符串并返回任务实例的工厂函数。
3. 调用工厂函数创建任务实例。
4. 根据类型决定将实例存入 NavTask 池还是 DaemonTask 池。

### 5.3 插件类型区分
- **NavTask（导航任务）**：参与主任务状态机流转，可被任务切换逻辑调度，同一时间只有一个 NavTask 处于活跃主任务位。
- **DaemonTask（守护任务）**：不参与主任务切换逻辑，常驻于守护任务栈，每帧被统一驱动，适合异常监控、安全守护等需要与主任务并行运行的逻辑。

### 5.4 容错设计
按名称查找任务时，若找不到指定任务会自动回退返回 IdleTask，防止因插件加载失败导致调度器崩溃。

## 6. 异常处理与状态回退

异常处理分布在状态监测和命令响应两个层面。

### 6.1 硬件异常监测
在 robotStateUpdate 中，若 NavBridge 上报非零错误码，系统按位解析各类异常。异常发生时机器人全局状态被置为错误，当前异常类型被记录，供守护任务在后续帧中读取并执行保护动作。

### 6.2 低电量回退
当电量低于阈值时，系统自动设置回充标志。在 hackHandle 中检测到该标志后，停止当前主任务，启动回充任务并追加到守护任务栈，机器人状态置为回充中。

### 6.3 外部停止与回退
- 作业完成标志：直接切回 IdleTask。
- 清扫标志：从待机状态进入清扫准备流程。
- 任务自身失败：如区块清扫超时或规划器不可用，子类在核心循环中设置失败标志，将下一任务指向 IdleTask，然后调用 Stop，由调度器在下一帧自动切换到待机。

## 7. TCP 文本命令与 RPC 处理

当前实现中提供了一套基于 TCP 文本协议的远程控制接口。

### 7.1 服务端架构
RPC 在独立线程中运行经典的阻塞式 Socket 服务器，使用 select 实现带超时的连接监听，确保线程能响应退出信号。每接受一个客户端连接，读取一行文本命令，解析后返回文本响应并关闭连接。

### 7.2 命令集
支持的文本指令包括：
- **查询类**：返回电量、充电状态、位姿、错误码、状态机、当前任务名的键值对字符串。
- **控制类**：开始/停止清扫、暂停、恢复、停止、回充、遥控速度下发、基站命令（洗拖布、烘干、集尘等）。

### 7.3 与调度器的数据交换
- **上行**：每周期将机器人数据和状态机写入 RPC 缓存，供状态查询实时读取。
- **下行**：命令处理函数直接修改 NavBridge 中的标志位（如暂停、清扫、回充等），调度器在下一周期的 hackHandle 中感知并执行相应动作。

---

## 8. 整体数据流

```
LCM 消息总线 → NavBridge（数据聚合 + TF + 规划器）
                     ↓
              TaskScheduler（20ms 锁保护循环）
                     ↓
      ┌──────────────┼──────────────┐
      ↓              ↓              ↓
 cleanTaskUpdate  deamonTaskUpdate  robotStateUpdate
      ↓              ↓              ↓
  BaseTask_t      DaemonTask      异常/电量判断
      ↓                            ↓
 MainLoop()                   hackHandle()
      ↓                            ↓
   规划器控制                   任务切换/暂停/回充
```

navclean 进程本质上是一个固定周期、单主任务加多守护任务并发、基于消息总线驱动的实时调度系统。所有任务通过 NavBridge 共享外部世界的状态，通过统一的生命周期接口实现可插拔的行为组合。
