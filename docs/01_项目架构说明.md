# FJ212 Alpha 项目架构说明文档

> 本文档从总体到局部，系统阐述 FJ212 Alpha 扫地机器人导航清扫软件的总体架构、各组件职责、组件间的通信方式与交互协议。文档不含源码细节，面向架构理解与系统维护。

---

## 1. 总体概述

### 1.1 项目定位

FJ212 Alpha 是一款扫地/拖地机器人的嵌入式导航清扫系统软件，负责机器人的**实时定位、地图构建、路径规划、任务调度、人机交互、云端对接**等全链路功能。

### 1.2 技术特征

| 维度 | 说明 |
|------|------|
| 开发语言 | C++17 |
| 构建系统 | CMake 3.16+ |
| 运行平台 | 嵌入式 Linux（ARM64，RK3576 / RK3566） |
| 架构风格 | 多进程分布式 + 插件化任务调度 + 发布-订阅消息总线 |
| 核心算法 | GMapping（栅格 SLAM）、AMCL（蒙特卡洛定位）、D* Lite（路径规划） |

### 1.3 设计原则

- **进程隔离**：感知、规划、建图、定位、交互、云端对接按职责拆分为独立进程，单点故障不影响全局。
- **插件扩展**：导航任务以动态库（`.so`）形式加载，支持运行时扩展新任务类型，无需重编主进程。
- **消息驱动**：进程间以 LCM（Lightweight Communications and Marshalling）消息总线为核心数据通道，解耦生产者与消费者。
- **分层抽象**：硬件细节封装在平台库，算法封装在独立库，上层仅通过标准接口调用。

## 2. 系统分层架构

整个系统自上而下分为六个层次：

```
┌─────────────────────────────────────────────────────────────┐
│  应用进程层 (Apps)                                           │
│  navclean │ hmi_server │ slam │ localization │ app_server   │
│  monitorServer │ daemon                                      │
├─────────────────────────────────────────────────────────────┤
│  任务插件层 (Plugins)                                        │
│  back_to_dock │ map_explorer │ relocalization │ dstar_planner│
│  exception_handler │ vir_bumper │ base_station │ daemon_base_station│
├─────────────────────────────────────────────────────────────┤
│  核心算法库层 (Libs)                                         │
│  navtask │ slam │ localization                               │
├─────────────────────────────────────────────────────────────┤
│  硬件抽象与工具库层                                          │
│  platform (HAL) │ xutils (基础设施)                          │
├─────────────────────────────────────────────────────────────┤
│  第三方依赖层 (Third Party)                                  │
│  lcm │ opencv │ eigen3 │ gmapping                             │
├─────────────────────────────────────────────────────────────┤
│  硬件与外部系统层                                            │
│  激光雷达 │ 轮式里程计 │ IMU │ MCU │ 充电基站 │ 云端/App    │
└─────────────────────────────────────────────────────────────┘
```

### 2.1 应用进程层

由 7 个独立可执行进程组成，每个进程对应一个系统职责，运行时通过 LCM 总线协同。

### 2.2 任务插件层

由 8 个动态库组成，在 `navclean` 进程启动时通过 `dlopen` 加载。插件分为两类：
- **NavTask 插件**：承担主任务逻辑（如回充、探索、重定位）。
- **DaemonTask 插件**：承担守护/监控逻辑（如异常处理、虚拟碰撞、基站守护），通常以 Hook 形式挂载到主任务中。

### 2.3 核心算法库层

- **navtask**：导航任务调度框架，含任务状态机、规划器封装、TF 坐标变换、NavBridge 资源聚合。
- **slam**：基于 GMapping 的二维栅格 SLAM，含地图后处理、房间分割、RPC 服务。
- **localization**：基于 AMCL 的自适应蒙特卡洛定位，含粒子滤波、多传感器模型。

### 2.4 硬件抽象与工具库层

- **platform**：硬件抽象层（HAL），直接操作串口、激光雷达、MCU，将原始传感器数据封装为结构化消息。
- **xutils**：全系统基础设施，提供 LCM 消息定义、日志、配置解析、时间同步滤波、RPC 框架、TCP 基础封装。

### 2.5 第三方依赖层

- **lcm**：进程间发布-订阅通信库，ARM 目标板使用 UDP 多播，x86 开发机使用空实现桩。
- **opencv**：地图图像处理、房间分割、地图后处理。
- **eigen3**：矩阵运算、TF 坐标变换。
- **gmapping**：FastSLAM 算法库，被 slam 库依赖。

## 3. 应用进程层详解

### 3.1 navclean（主导航清扫进程）

**定位**：整机任务调度中心，系统最核心的控制进程。

**职责**：
- 启动时动态加载所有任务插件（`.so`）。
- 以约 20ms 为周期循环调度当前主任务（如房间清扫、回充、探索）和守护任务（异常处理、虚拟碰撞、基站守护）。
- 通过 `NavBridge` 聚合所有外部资源（底盘、规划器、地图服务、语音、TF 变换等），供插件统一访问。
- 维护机器人全局状态：电量、充电状态、碰撞/悬崖状态、清扫统计、房间/禁区数据等。
- 接收 HMI 的文本命令和外部 RPC 请求，转换为任务调度指令。

**核心线程**：
- 调度线程：20ms 周期主循环，驱动任务更新、状态检查、RPC 数据交换。

### 3.2 hmi_server（人机交互进程）

**定位**：人机交互中枢与状态仲裁器。

**职责**：
- 管理实体按键（电源键 / Fn 键）的检测与解码。
- 管理 LED 状态机（白/红/绿闪烁），反映机器人当前工作状态。
- 作为 RPC 服务端，接收 `app_server` 发来的云端/App 命令，进行状态仲裁后转发给 `navclean`。
- 通过 TCP 文本协议（端口 9001）向 `navclean` 发送命令（如 `clean`、`charge`、`pause`、`resume`、`stop`）。
- 通过 LCM 广播机器人状态，供 App 和监控端消费。

**核心线程**：
- 按键检测与 LED 更新线程。
- 主循环线程：周期性查询 `navclean` 状态、分发 App 命令、状态变化检测。

### 3.3 slam（建图进程）

**定位**：独立建图引擎。

**职责**：
- 订阅激光雷达与里程计数据，基于 GMapping 执行粒子滤波 SLAM，构建占据栅格地图。
- 支持地图保存/加载、房间自动分割、虚拟墙/禁区管理、地图对齐（多楼层）。
- 将建图结果（地图、房间块、位姿）通过 LCM 广播给定位、导航、App 等模块。
- 提供 RPC 服务（端口 9005），支持外部调用房间编辑、重定位指令等。

**核心线程**：
- 主建图循环线程：接收传感器、调用 GMapping 更新、地图发布。
- 事件处理线程：处理 SLAM 内部事件和 RPC 命令。
- GMapping 后台线程：粒子滤波与地图优化。

### 3.4 localization（定位进程）

**定位**：独立定位引擎。

**职责**：
- 基于 AMCL 算法，订阅 SLAM 地图、激光雷达、里程计数据，估计机器人在地图中的实时位姿。
- 支持初始位姿设置、重定位触发、粒子云发布。
- 将定位结果（位姿、里程计修正、粒子云）通过 LCM 广播给导航和建图模块。

**核心线程**：
- 定位主线程：激光/里程计融合、粒子滤波更新、重采样。
- 位姿发布线程：定时发布定位结果。

### 3.5 app_server（云端/App 代理进程）

**定位**：云端 IoT 平台的本地代理。

**职责**：
- 对接涂鸦（Tuya）IoT SDK，接收来自云端的 DP（Data Point）命令，转换为机器人本地任务命令。
- 反向上报机器人状态、地图、清扫轨迹、固件信息、故障码等给云端。
- 作为 RPC 客户端，将 App 命令转发给 `hmi_server`（端口 9002）。
- 订阅 LCM 上的地图、轨迹、状态消息，进行压缩和封装后上报。

**核心线程**：
- App 任务线程：读取 Tuya SDK 命令，通过 RPC 转发给 HMI。
- Tuya SDK 线程：维持云端长连接。

### 3.6 monitor_server（监控服务进程）

**定位**：可视化与调试数据聚合服务器。

**职责**：
- 订阅导航、建图、传感器等各类 LCM 消息，缓存最新数据。
- 通过 TCP TLV 协议（端口 8989）响应外部监控客户端的请求。
- 支持的数据类型包括：地图、全局/局部代价图、激光点云、里程计、路径、粒子云、房间形状等。

**核心线程**：
- 主服务线程：处理 LCM 消息队列 + 读取 TCP 客户端请求 + 编码发送对应数据。

### 3.7 daemon（日志守护进程）

**定位**：极简系统守护。

**职责**：
- 按配置周期调用 `logrotate` 进行日志轮转。
- 无业务逻辑，不介入任何 LCM 或网络通信。

## 4. 核心库层详解

### 4.1 navtask（导航任务核心库）

**定位**：导航系统的"大脑"框架。

**核心概念**：
- **BaseTask_t**：所有任务的抽象基类，定义了统一的生命周期钩子（`PreStart`、`PreStop`、`PreResume`、`PreSuspend`、`MainLoop`）。
- **TaskScheduler_t**：顶层调度器，维护主任务和守护任务的运行状态，20ms 周期驱动。
- **NavBridge_t**：中央桥梁对象，所有任务通过它访问外部资源（底盘、规划器、地图服务、语音、TF 树等），避免任务直接依赖底层硬件。
- **TaskPools_t**：任务工厂，管理内置任务和动态加载的插件任务。
- **TF_t / Frame_t**：基于 Eigen3 的 2D/3D 坐标变换库，维护 Map → Odom → Base 的 TF 树。

**任务类型**：
内置支持 18+ 种清扫/导航任务，包括房间清扫、选区清扫、定点清扫、沿墙清扫、回充导航、路径跟随、遥控、Idle 等。

### 4.2 slam（建图库）

**定位**：GMapping 的封装与扩展。

**核心概念**：
- **Slam_t**：SLAM 主类，双线程架构（主循环 + 事件处理）。
- **SlamRpc_t**：SLAM 的 RPC 服务桩，对外暴露房间编辑、重定位等接口。
- **MapProcess / segment_room**：地图后处理与房间分割，支持 Voronoi 分割、形态学分割、距离分割、波前区域生长等算法。
- **Map_align**：地图对齐与旋转校正。

### 4.3 localization（定位库）

**定位**：AMCL 算法的完整实现。

**核心概念**：
- **Amcl_t**：AMCL 主类，双 LCM 实例（接收 / 发布）+ 双线程。
- **pf_t**：粒子滤波器核心，维护双样本集，支持选择性重采样、K-D 树聚类、收敛性检测。
- **AMCLLaser**：激光传感器模型，支持光束模型、似然场模型、概率似然场模型（含 beam skipping）。
- **AMCLOdom**：里程计动作模型，支持差分/全向/修正差分模型。

### 4.4 platform（硬件抽象层）

**定位**：连接物理世界的唯一通道。

**核心概念**：
- **XBase_t**：硬件抽象核心类，管理整个传感器与执行器栈：激光雷达启动/关闭、MCU 数据解析、里程计滤波、IMU/Yaw 处理、虚拟碰撞、悬崖检测、3D 障碍物检测、地毯检测、拖地/主刷升降、速度控制、轨迹记录、清扫面积计算、按键处理、电量平滑与打滑检测等。
- **XStation_t**：基站通信抽象，处理与清洗/充电基站的 RF 数据交换。
- **xprotocol.h**：定义 MCU 通信二进制协议，含帧头/帧尾、电机 PWM、编码器、ADC、IMU 四元数、RF 数据包等。

### 4.5 xutils（基础设施库）

**定位**：全系统的公共底座。

**核心概念**：
- **XLog**：分级日志系统（FATAL / ERROR / WARN / INFO / DEBUG）。
- **ConfigFile / XCfg**：配置文件读写器，支持多种数据类型。
- **TimeFilter**：时间同步模板类，用于多传感器（激光、里程计、IMU）的时间对齐。
- **LCM 消息定义（Msg/）**：按命名空间组织全系统约 50+ 种消息类型，是跨进程通信的契约。
- **Rpc 封装（Rpc/）**：内嵌 `rest_rpc` 头文件库（基于 `asio` + `msgpack-c`），提供跨进程 RPC 能力。
- **Socket（Socket/）**：极简 C 风格 TCP 服务端/客户端封装。

## 5. 插件系统详解

### 5.1 加载机制

所有插件编译为共享动态库（`.so`），存放在 `lib/plugins/` 目录下。`navclean` 启动时通过 `dlopen` + `dlsym` 动态加载，入口符号为统一的 `CreatClass` 函数。

### 5.2 插件分类

| 插件 | 类型 | 功能概述 |
|------|------|----------|
| **back_to_dock** | NavTask | 回充导航与对接。状态机：加载充电桩位姿 → 导航 → 对齐 → 充电 → 结束。 |
| **map_explorer** | NavTask | 地图探索与建图。搜索地图 frontier（边界）→ 导航前往 → 检查完成度，最多循环 3 次。 |
| **relocalization** | NavTask | 重定位触发与监控。管理重定位标志位，依赖外部 SLAM/AMCL 完成实际计算。 |
| **exception_handler** | DaemonTask | 异常/错误监控。每帧检查错误标志，通常作为 Hook 挂载到主任务中。 |
| **vir_bumper** | DaemonTask | 3D 障碍物虚拟碰撞检测。检查 3D 障碍物数据，输出虚拟碰撞触发标志。 |
| **base_station** | NavTask | 基站内状态同步。同步基站相关配置和充电标志，维持机器人在基站状态。 |
| **daemon_base_station** | DaemonTask | 基站守护占位。轻量级任务，仅在离开基站时标记状态。 |
| **dstar_planner** | 规划器 | D* Lite 路径规划器。基于优先队列的增量式搜索，8 邻域，带墙距惩罚。 |

### 5.3 插件接口与生命周期

所有 NavTask / DaemonTask 插件继承自 `BaseTask_t`，必须实现以下生命周期钩子：

- **PreStart**：任务启动前初始化。
- **PreStop**：任务停止前清理。
- **PreResume**：任务恢复前准备。
- **PreSuspend**：任务暂停/挂起时处理。
- **MainLoop**：任务运行期间每 20ms 调用一次（仅在 RUNNING 且非 suspend 状态）。

Hook 机制允许守护任务以 `mainHookStack` 形式挂载到主任务中，每帧在主逻辑前执行，实现低侵入式监控。

## 6. 通信架构详解

系统采用 **多协议混合通信架构**，不同协议承担不同职责：

| 协议 | 模式 | 用途 | 典型数据 |
|------|------|------|----------|
| **LCM** | Pub/Sub（UDP 多播） | 进程间实时数据流 | 传感器、地图、位姿、轨迹、状态 |
| **RPC** | Client/Server（TCP + msgpack） | 进程间命令/配置/查询 | 房间编辑、重定位、App 指令、参数设置 |
| **TCP Socket** | Client/Server | 外部工具调试 | 监控可视化、遥控、地图拉取 |
| **UART** | 主从帧协议 | 底层硬件通信 | MCU 指令、激光雷达原始数据、RF 基站 |

### 6.1 LCM 消息总线

LCM 是系统的核心通信骨架，承担所有进程间的实时数据交换。

#### 6.1.1 运行时策略

- **ARM 目标板**：使用真实的 LCM 运行时，基于 UDP 多播通信。
- **x86 开发机**：使用空实现桩（stub），所有 API 为空，仅保证编译通过，不依赖 GLib2 运行时。

#### 6.1.2 消息定义

所有 LCM 消息定义位于 `config/lcm/`，按功能分包为 9 个命名空间：

| 包名 | 功能域 | 典型消息 |
|------|--------|----------|
| NavMsg | 导航与地图 | VelCmd、Odom、Pose、GridMap、BlockArray、SlamCmd |
| SensorMsg | 传感器 | LaserData、ImuData、Bumper、Cliff、WallSensor、PointCloud |
| RobotMsg | 底层状态与控制 | RobotStatus、Power、MotorDutyCmd、HackCmd |
| AppMsg | App 层指令 | CleanCmd、DutyCmd、AppCmd |
| TuyaMsg | 涂鸦 IoT | TuyaXtPDO、TaskFinishStatus、VirWallAndPolygen |
| RpcMsg | RPC 状态同步 | RobotData、RobotInfoData、RobotParaData |
| UtilsMsg | 通用工具 | TimerSync、RpcCmd |
| ProMsg | 进程管理 | StateMsg |
| CostMap2DMsg | 代价图 | PointCloud |

#### 6.1.3 通道命名规范

所有 LCM 通道名统一以宏形式定义在全局头文件中，宏前缀为 `LCM_CHANNEL_`。命名风格为 PascalCase 字符串，如 `"VelCmd"`、`"LaserScan"`、`"BlockInfoFromSlam"`。

#### 6.1.4 核心通信矩阵

以下为各进程/库在 LCM 总线上的主要收发关系：

**platform / XBase（硬件中枢）**
- 发布：LaserScan、Imu、Odom、Bumper、Cliff、WallSensor、Power、RobotStatus 等全部传感器数据。
- 订阅：VelCmd（速度指令）、MotorDutyCmd（电机占空比）、HairCutCmd、HackCmd、DutyCmd 等控制指令。

**slam（建图）**
- 发布：GmMap（栅格地图）、SlamMapInfo（地图信息）、BlockArray（房间块）、AppMap（App 用压缩地图）、SlamPoseInfo、粒子云、DistrictShape 等。
- 订阅：LaserScan、Odom、AmclOdom、SlamCmd（建图命令）、BumperForSlam、HackCmd、RobotStatus、RpcCmd、TimerSync 等。

**localization（定位）**
- 发布：AmclPose（带协方差的定位位姿）、AmclOdom、AmclParticleCloud、AmclPoseInfo、AmclToSlamCmd 等。
- 订阅：LaserScan、Odom、GridMap / GridMapInfo（来自 SLAM）、InitialPoseCmd（初始位姿设置）、HackCmd、RobotStatus、RpcCmd、TimerSync 等。

**navtask / NavBridge（导航中枢）**
- 发布：VelCmd（速度指令）、InitAmclPose（初始位姿下发）等。
- 订阅：Odom、AmclOdom、Power、Bumper、RobotStatus、SlamMapInfo、BlockInfoFromSlam、AmclPoseInfo、VelCmd 等。

**hmi（人机交互）**
- 发布：Hmi_Broadcast_RobotState（机器人状态广播）、hmi_publish_vel_nav（遥控速度）、hmi_publish_para_app（参数上报）等。
- 订阅：NavPro_publish_RobotInfo（导航状态）、MapInfo、HackCmd 等。

**app_server（云端代理）**
- 订阅：AppMap、AppTraj（轨迹）、Hmi_Broadcast_RobotState、hmi_publish_para_app、slam_publish_para_app、BlockInfoFromSlam 等。
- 无 LCM 发布，通过 RPC 与 HMI 交互。

**monitor_server（监控服务）**
- 订阅：DstarMap、GlobalCostMap、LocalCostMap、LaserScan、Odom、GmOdom、AmclOdom、AmclParticleCloud、GlobalPath、LocalPath、PredPath、DistrictShape、FastCostMap、mapexp3Dobs 等大量调试数据。
- 无 LCM 发布，通过 TCP TLV 向 PC 监控端转发。

**teleop（遥控工具）**
- 发布：VelCmd、HackCmd。


### 6.2 RPC 框架

RPC 用于进程间的**请求-响应式命令/控制通信**，与 LCM 的**发布-订阅式数据流**互补。

#### 6.2.1 技术栈

- 网络层：`asio`（内嵌头文件库）
- 序列化：`msgpack-c`（内嵌头文件库）
- 框架：`rest_rpc`（内嵌头文件库，位于 `xutils`）

#### 6.2.2 端口分配

| 进程 | 角色 | 端口 |
|------|------|------|
| hmi_server | RPC Server | 9002 |
| app_server | RPC Client → HMI | 9003（对外）/ 连 9002 |
| navclean | RPC Server | 9004 |
| slam | RPC Server | 9005 |

#### 6.2.3 典型 RPC 交互

- **app_server → hmi_server**：App 端通过涂鸦 SDK 收到云端指令后，作为 RPC Client 调用 HMI 的 `SetAppCmd`、`RpcApi_APPCMD` 等接口，由 HMI 仲裁后转发给 navclean。
- **外部工具 → slam**：通过端口 9005 调用 `SlamRpc_t`，执行房间编辑（添加/删除虚拟墙、禁区、房间分割/合并/命名）、重定位指令等。
- **外部工具 → navclean**：通过端口 9004 调用 `TmRpc_t`，查询机器人状态或下发控制指令。

### 6.3 TCP Socket

#### 6.3.1 monitor_server TCP 服务（端口 8989）

`monitor_server` 作为 TCP Server，接收上位机或 `monitor_client` 的连接。协议采用 **TLV（Type-Length-Value）** 格式，客户端发送 `monitor_request_t` 请求，服务端返回对应的结构化数据。

支持的数据请求类型：
- MAP（栅格地图）
- LASER（激光点云）
- ODOM（里程计）
- PATH（全局/局部路径）
- COSTMAP（代价图）
- PARTICLE（粒子云）
- BLOCK（房间块）

#### 6.3.2 monitor_client / monitor_client_cli

- **monitor_client**：带 ROS 桥接功能的 GUI 可视化工具。通过 TCP 从 `monitor_server` 拉取数据，转换为 ROS Topic 和 TF 广播，供 RViz 可视化。ROS 为可选依赖。
- **monitor_client_cli**：无 ROS 依赖的命令行工具，直连 `monitor_server:8989`，打印接收到的数据元信息，可将地图帧保存为本地二进制文件。

#### 6.3.3 HMI ↔ navclean TCP 文本协议（端口 9001）

`hmi_server` 作为 TCP Client，主动连接 `navclean:9001`，发送纯文本命令。命令集包括：`clean`、`charge`、`pause`、`resume`、`stop`、`status` 等。这是一种轻量级的内部控制通道。

#### 6.3.4 XD1Y 激光调试接口（端口 8888）

用于 XD1Y 结构光/ToF 传感器的原始数据调试。

### 6.4 UART 串口通信

UART 是系统与底层硬件的唯一物理通道。

#### 6.4.1 MCU 通信

`XBase_t` 通过串口（`/dev/ttyS6` 或 `/dev/ttyS3`，依硬件平台而定）与底盘 MCU 通信。协议为自定义二进制帧格式：

- 帧结构：帧头 + 命令字 + 数据域 + 校验 + 帧尾。
- 上行（MCU → 主机）：电机编码器、ADC 采样、碰撞/悬崖传感器状态、IMU 四元数、电量、RF 基站数据包等。
- 下行（主机 → MCU）：电机 PWM 占空比、风机转速、拖地升降、主刷升降、LED 控制、蜂鸣器等。

#### 6.4.2 激光雷达接入

- **X1C / XD1Q**：通常通过 UART 或网口接入，由 `XBase_t` 驱动管理。
- **XD1Y**：通过 I2C/UART 接入，由独立的 `xd1y` 驱动模块处理原始数据并生成点云。

## 7. 典型业务流程

### 7.1 全屋清扫任务流程

1. **触发**：用户通过 App 下发"全屋清扫"指令，经涂鸦云端到达 `app_server`。
2. **仲裁**：`app_server` 通过 RPC 调用 `hmi_server` 的 `SetAppCmd` 接口。
3. **转发**：`hmi_server` 状态仲裁通过后，通过 TCP 文本协议向 `navclean:9001` 发送 `clean` 命令。
4. **调度**：`navclean` 的 `TaskScheduler_t` 识别命令，启动对应的 `RoomCleanTask`（或从插件加载对应任务）。
5. **规划执行**：任务通过 `NavBridge_t` 访问 `ZPlanner` / `DStarPlanner` 生成全局路径，通过 `PurePursuit` 跟踪局部路径。
6. **控制下发**：`navclean` 通过 LCM 发布 `VelCmd`，`XBase_t` 订阅后转换为电机 PWM 下发给 MCU。
7. **状态闭环**：
   - `XBase_t` 持续发布 LaserScan、Odom 到 LCM。
   - `localization` 订阅后计算 AmclPose，发布定位结果。
   - `slam` 可选地订阅传感器数据更新地图（若处于建图模式）。
   - `navclean` 订阅 AmclPose 进行定位反馈控制，同时监控 Bumper、Cliff、Power 等状态。
8. **守护任务**：`exception_handler`（Hook 挂载）每 20ms 检查错误标志；`vir_bumper` 检查 3D 障碍物；若有异常则触发暂停/停止。
9. **完成上报**：清扫完成后，`navclean` 通过 LCM 发布任务完成状态，`hmi_server` 更新 LED 状态，`app_server` 上报云端。

### 7.2 回充任务流程

1. **触发**：用户 App 点击"回充"，或电量低自动触发，或清扫完成自动回充。
2. **命令链路**：App → `app_server` → `hmi_server`（RPC）→ `navclean`（TCP:9001）。
3. **任务切换**：`navclean` 停止当前主任务，启动 `back_to_dock` 插件任务。
4. **回充导航**：
   - `back_to_dock` 从 `NavBridge` 读取充电桩位姿（`chargePoints` 或 `stationSlamPose`）。
   - 调用 `DStarPlanner` 规划到充电桩的全局路径。
   - 机器人沿路径行驶至充电桩附近，进入对齐阶段。
5. **对接充电**：对齐完成后，`XBase_t` 检测到充电信号，`back_to_dock` 设置 `robotIsCharging = true`，任务结束。
6. **状态同步**：`base_station` 任务（若在基站内）同步基站相关状态；`daemon_base_station` 在离开基站时标记 `robotInStation = false`。

### 7.3 建图流程

1. **触发**：用户 App 下发"快速建图"或首次使用自动进入建图模式，`hmi_server` 转发 `slam` 相关命令。
2. **SLAM 启动**：`slam` 进程已常驻运行，收到 `SlamCmd` 后进入建图模式。
3. **数据融合**：`slam` 订阅 `LaserScan` 和 `Odom`，在 `mThread` 中调用 `GridSlamProcessor` 执行粒子滤波 SLAM。
4. **地图输出**：
   - 实时发布 `GmMap`（栅格地图）给 `localization` 用于定位。
   - 发布 `SlamMapInfo`、`BlockArray`（房间块）给 `navclean` 和 `app_server`。
5. **探索辅助**：若启用探索建图，`navclean` 加载 `map_explorer` 插件，驱动机器人主动寻找地图 frontier（未知区域边界）。
6. **地图后处理**：建图完成后，`slam` 使用 OpenCV 进行地图优化（膨胀、滤波），调用 `segment_room` 进行房间自动分割。
7. **保存与上报**：用户通过 App 调用 `SlamRpc_t` 的保存地图接口，地图持久化到本地，房间信息上报云端。

### 7.4 重定位流程

1. **触发**：机器人开机、被搬动、定位丢失（粒子云发散）时，由 `hmi_server` 或 `navclean` 触发重定位命令。
2. **任务加载**：`navclean` 启动 `relocalization` 插件任务，设置 `relocalizationIsRunning = true`。
3. **定位初始化**：
   - `localization` 收到 `InitialPoseCmd`（来自 `navclean` 或 `slam`），在指定区域内重新初始化粒子分布。
   - 或 `slam` 通过 RPC 直接下发重定位指令，调整地图与当前观测的对齐。
4. **收敛检测**：`localization` 的 `pf_t` 通过 K-D 树聚类和协方差检测判断粒子云是否收敛。
5. **完成**：定位收敛后，`localization` 发布稳定的 `AmclPose`，`relocalization` 任务设置 `relocalizationRunningFinish = true` 并退出，`navclean` 恢复正常清扫任务。

## 8. 构建与部署

### 8.1 构建系统

- **构建工具**：CMake 3.16+
- **C++ 标准**：C++17
- **编译数据库**：自动生成 `compile_commands.json`，供 `clangd` 代码补全与跳转使用。
- **双平台构建**：
  - `build_rk3576/`：交叉编译（AArch64，生成机器人目标二进制）。
  - `build_x86/`：本机编译（x86，用于本地开发、单元测试、CI）。

### 8.2 构建顺序与依赖层级

```
third_party (lcm → opencv → eigen3)     ← 底层第三方库
    ↓
libs/xutils  (SHARED)                    ← 基础设施与消息定义
    ↓
third_party/gmapping (fastSlam)          ← SLAM 算法库
    ↓
libs/platform  (STATIC)                  ← 硬件抽象层
    ↓
libs/slam      (SHARED)                  ← 建图库
    ↓
libs/localization (SHARED)               ← 定位库
    ↓
libs/navtask   (STATIC)                  ← 导航任务核心（内嵌 DStarLite）
    ↓
plugins        (8 个 SHARED 插件 .so)     ← 动态任务插件
    ↓
apps           (7 个可执行进程)           ← 业务进程
    ↓
tools / tests  (可选)                    ← 调试工具与测试
```

### 8.3 输出目录结构

构建产物统一输出到构建目录下的 `bin/`、`lib/`、`plugins/`：

- **`bin/`**：可执行进程（`navclean`、`hmi_server`、`slam`、`localization`、`appTask`、`monitorServer`、`daemon`、`teleop` 等）。
- **`lib/`**：静态库（`.a`）与共享库（`.so`），包括 `libxutils.so`、`libslam.so`、`liblocalization.so`、`liblcm.so`（ARM）/ `liblcm.a`（x86 stub）等。
- **`lib/plugins/`**：8 个任务插件动态库（`.so`）。

### 8.4 运行时部署

机器人端启动顺序由 `startFj212.sh` 脚本控制：

1. `daemon` —— 日志守护。
2. `navclean` —— 任务调度中心。
3. `slam` —— 建图引擎。
4. `localization` —— 定位引擎。
5. `appTask` —— 云端代理（可选）。
6. `hmi_server` —— 人机交互。
7. `monitorServer` —— 监控服务（可选）。

脚本同时配置 LCM 多播路由、核心转储、网络缓冲区调优。

### 8.5 配置文件与资源

- **`config/`**：LCM 消息定义、日志配置、启动脚本。
- **`config/voice/`**：118 种语音资源（`.wav`），对应回充、充电完成、定位中、故障提示、清洗拖布、WiFi 配对等场景。

## 9. 总结

FJ212 Alpha 采用**多进程分布式 + 插件化 + 消息总线**的架构设计：

- **进程隔离**保证了系统的稳定性与可维护性，感知、建图、定位、规划、交互各司其职。
- **插件化**赋予了任务系统高度灵活性，新任务类型可通过动态库热插拔，无需重编主进程。
- **LCM 消息总线**作为实时数据骨架，以发布-订阅模式解耦了传感器、算法、应用各层。
- **RPC + TCP Socket** 作为命令与调试通道，补充了 LCM 在请求-响应和外部工具对接方面的能力。
- **UART 串口**作为唯一硬件通道，将复杂的 MCU 协议封装在平台库内，向上层提供干净的数据接口。

这一架构既满足了嵌入式实时系统的性能要求，又兼顾了功能扩展与工程维护的便利性。
