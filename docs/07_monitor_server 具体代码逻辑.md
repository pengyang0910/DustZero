# monitor_server 具体代码逻辑

> 本文档描述 monitor_server（监控服务进程）的内部代码逻辑，包括启动流程、单线程主循环、LCM 消息订阅与缓存、TCP TLV 协议、客户端连接管理，不含源码。

---

## 1. 总体架构

monitor_server 是一个数据可视化中继服务进程，核心职责是：订阅机器人系统内各模块通过 LCM 发布的导航与传感器数据，缓存最新数据，并通过 TCP 协议以 TLV 格式响应外部客户端的拉取请求。该进程采用单线程主循环模型，同时集成了 LCM 消息处理与 TCP 服务。

---

## 2. 启动流程

### 2.1 CPU 核心绑定
进程启动后，首先将自身线程绑定到预定义的 CPU 核心，避免在不同核心间迁移带来的缓存失效和调度抖动。

### 2.2 ViewServer_t 实例化
创建核心类的实例对象。在构造函数中完成以下初始化：
- 读取配置文件 monitor.cfg，获取日志开关和日志级别。
- 初始化日志模块，创建日志文件。
- 将所有数据更新标志位初始化为 false。
- 将运行标志设为 true。

### 2.3 启动与阻塞等待
调用实例的 start 方法触发后台工作线程，然后主线程阻塞在工作线程上，等待其结束。

在 start 方法中，依次完成：
- **LCM 回调注册**：调用设置消息回调函数，将所有需要的 LCM 通道与成员回调函数绑定。
- **工作线程创建**：创建一个名为 Monitor 的线程，线程入口为核心主循环函数，并绑定到相同的 CPU 核心。

---

## 3. 核心线程模型与主循环逻辑

monitor_server 唯一的工作线程承担了三项核心职责：LCM 消息消化、TCP 客户端连接管理、请求响应与数据发送。

### 3.1 循环体结构
主循环按以下流程执行：

1. **LCM 消息处理**：调用 LCM 处理函数。该函数内部是一个小循环，以 1 毫秒超时反复调用 LCM 超时处理，直到没有新消息为止。这意味着每次主循环会将当前 LCM 队列中积压的消息全部消费掉。

2. **客户端连接检测**：
   - 如果当前没有客户端连接，非阻塞地检测是否有新的 TCP 连接请求。如果有，则接受连接并设置新客户端套接字。
   - 如果已有客户端连接，先检测连接是否已断开（通过查询 TCP 连接状态）。

3. **请求读取与响应**：
   - 若连接有效，以极短超时尝试读取 1 个字节的请求命令。
   - 若读到数据，根据该字节的枚举值进入对应的分支处理。
   - 若未读到数据，休眠 5 毫秒后继续下一次循环。

4. **无连接时的休眠**：如果始终没有客户端连接，主循环休眠 40 毫秒，降低 CPU 占用。

### 3.2 整体线程特征
- **单线程串行模型**：所有 LCM 消息处理、TCP 监听、数据编码和发送都在同一个线程内串行完成，没有额外的 LCM 专属线程。
- **非阻塞 I/O**：TCP 服务端和客户端套接字均被设置为非阻塞模式，通过 select 实现超时检测。
- **时间片驱动**：主循环天然是一个时间片轮询结构，即使没有任何客户端，也会持续运行以消化 LCM 消息。

## 4. LCM 消息订阅与缓存

### 4.1 订阅的通道列表
设置消息回调函数方法注册了多达 17 个 LCM 通道的订阅。每个订阅都绑定到成员回调函数。部分关键订阅设置了队列容量为 1，丢弃旧消息以保证实时性。

订阅的通道覆盖以下数据类型：
- **地图类**：DstarMap（规划器地图）、GlobalCostMap（全局代价地图）、LocalCostMap（局部代价地图）。
- **激光与点云类**：LaserScan（激光雷达数据）、LaserXD1Q（XD1Q 激光点云）、LaserX1C（X1C 激光点云）、FastCostMap（FastSLAM 原始地图点云）、mapexp3DobsUpdate（3D 障碍物标记）。
- **里程计类**：Odom（底盘里程计）、GmOdom（SLAM 里程计）、AmclOdom（AMCL 定位后里程计）。
- **位姿数组类**：AmclParticleCloud（AMCL 粒子云）、slamParticleCloud（SLAM 粒子云）、NextLine（下一条清扫线）、odomNextLine（里程计坐标系下的下一条线）、GlobalPath（全局规划路径）、LocalPath（局部规划路径）、PredPath（预测路径）、DistrictShape（区域形状）。

### 4.2 回调函数的行为模式
所有回调函数遵循统一的简单模式：
- 将接收到的 LCM 消息直接赋值给对应的成员变量（拷贝）。
- 将对应的布尔标志位设为 true，表示该数据已更新。

例如，激光数据收到后，将激光数据缓存并设置激光更新标志。地图更新有一个特殊判断：只有当消息的 caller 字段等于 Monitor 时，才会接受该地图数据并更新标志。

## 5. TCP TLV 协议详解

### 5.1 服务端监听与连接管理
- **监听端口**：固定为 8989。
- **套接字创建**：创建 IPv4 TCP 流式套接字，绑定到任意地址，监听队列长度为 5，并启用地址复用。
- **接受连接**：使用 select 以零超时检测连接请求。收到连接后完成 accept，新客户端套接字被设为非阻塞模式，并禁用 Nagle 算法、设置 512KB 的发送缓冲区。
- **连接断开检测**：通过查询 TCP 信息中的连接状态字段，若状态不是已建立，则关闭套接字并将客户端句柄重置。

### 5.2 TLV 格式定义
服务器与客户端之间的数据包采用 TLV（Type-Length-Value）格式：
- **Type**：2 字节，大端序无符号整数，表示数据类型。
- **Length**：4 字节，大端序无符号整数，表示后续 Value 字段的字节长度。
- **Value**：变长字节数组，实际载荷数据。

总包头固定为 6 字节。

### 5.3 支持的请求类型与响应逻辑
客户端发送的请求仅为 1 个字节的命令码。服务器主循环中通过分支处理，支持以下请求：

| 10 | 全局代价图请求 | 同上，使用全局代价地图数据。 |

| 3 | AMCL 里程计请求 | 同上，使用 AMCL 里程计数据。 |
| 4 | AMCL 粒子云请求 | 同上，使用 AMCL 粒子云数据。 |

### 5.4 响应数据的发送方式
对于有效数据，服务器执行以下步骤：
1. 申请一块最大 100KB 的全局静态发送缓冲区。
2. 调用对应 LCM 消息类型的编码方法，将消息对象序列化为字节流写入缓冲区，返回实际长度。
3. 若数据长度合法（大于 0 且小于 100KB），调用写 TLV 函数发送完整的 TLV 包。
4. 若该类型数据无更新（标志位为假），则只发送 1 字节（值为 0）作为旧数据指示器，仍包装在 TLV 中。

## 6. 数据缓存与更新机制

### 6.1 内部数据结构
核心类内部为每种 LCM 数据类型维护了一个对应的成员变量作为缓存，同时维护一个布尔标志位表示是否有新数据：
- **地图类**：SLAM 地图、全局代价地图、局部代价地图、Dstar 规划器地图。
- **位姿数组类**：AMCL 粒子云、SLAM 粒子云、下一条清扫线、里程计坐标系下的下一条线、全局规划路径、局部规划路径、预测路径、区域形状、3D 障碍物标记。
- **里程计类**：底盘里程计、SLAM 里程计、AMCL 定位后里程计。
- **激光与点云类**：激光雷达数据、XD1Q 点云、X1C 点云、FastSLAM 原始地图点云。

### 6.2 更新与消费机制
- **更新端**：LCM 回调触发时，数据被直接拷贝到成员变量，标志位设为 true。
- **消费端**：TCP 主循环收到客户端请求时，检查标志位。若为新数据，调用编码方法序列化到发送缓冲区，同时将标志位重置为 false。
- **旧数据指示**：如果标志位为 false（即自上次请求以来没有收到新的 LCM 消息），服务器仍会响应，但 TLV 的 Value 只有 1 字节且内容为 0，客户端据此判断数据未更新。

这种机制保证：
- 每个新数据最多被发送一次（单客户端场景下的精确去重）。
- 客户端不会阻塞等待，总是能在短时间内得到响应。

## 7. 客户端连接管理

### 7.1 单客户端模型
monitor_server 本质上是一个单客户端服务。它内部只维护了一个客户端套接字变量：
- 在没有客户端时，反复尝试 accept 新连接。
- 一旦接受了一个客户端，就不会再 accept 其他连接（直到当前连接断开）。
- 如果第二个客户端尝试连接，它会在内核的 TCP 监听队列中排队（队列长度 5），但服务器不会处理，直到当前连接断开并回到 accept 状态。

### 7.2 连接断开处理
断开检测有两种方式：
1. **主动检测**：每次主循环开始时，查询 TCP 状态。若发现非已建立状态，关闭套接字并置客户端句柄为无效。
2. **被动检测**：在发送函数中，如果发送失败，会将客户端句柄置为无效。

断开后的恢复：
- 服务端不会退出，主循环继续运行。
- 客户端句柄被重置后，下一次循环会重新进入连接请求检测等待新连接。
- 若暂时无客户端，服务端休眠 40 毫秒，保持低 CPU 占用。

## 8. 与 monitor_client / monitor_client_cli 的交互

### 8.1 monitor_client_cli（命令行调试工具）
monitor_client_cli 是一个不依赖 ROS 的轻量级 CLI 工具，用于直接验证 TCP 链路：
- **连接**：连接到服务器的 IP 和端口。
- **请求循环**：预定义一组感兴趣的请求码（如 MAP、LASER、ODOM、GLOBAL_PATH 等），按固定间隔逐个发送 1 字节请求。
- **读取响应**：读取 6 字节包头，解析 Type 和 Length，再读取完整的 Value。
- **旧数据处理**：若收到的 Value 长度为 1 且内容为 0，则识别为"无新数据"；否则打印收到的数据长度，并可选地将地图数据保存为二进制文件。

### 8.2 monitor_client（ROS 可视化桥接工具）
monitor_client 是一个更复杂的 ROS 节点，负责将 monitor_server 的数据转换为 ROS 话题发布到 RViz：
- **三线程架构**：
  1. 数据拉取线程：维护一个处理器对象列表，循环遍历。对每个启用的类型，若未更新，则通过 TCP 向服务器发送 1 字节请求，接收 TLV 响应，解码后存入内部 LCM 结构。
  2. 数据发布线程：将已更新的数据通过 ROS Publisher 发布，并发布 TF 变换树。
  3. 独立地图接收线程：通过另一个端口以压缩块协议接收多图层地图，与主 TCP 通道分离。
- **请求与解码流程**：调用请求命令函数向套接字写入 1 字节请求码；读取 TLV 响应并反序列化；将解码后的数据转换为 ROS 标准消息类型，标记为已更新。
- **配置驱动**：通过 monitor.cfg 配置文件决定启用哪些数据类型的拉取，以及服务器 IP 地址。

## 9. 关键设计特点

1. **简单高效的单线程模型**：LCM 消费、TCP 服务、数据编码全部在一个线程内完成，避免了多线程锁竞争。
2. **拉取式数据分发**：服务器不会主动推送数据，完全依赖客户端请求触发发送，天然支持客户端按需订阅。
3. **新旧数据去重**：通过布尔标志位实现"每个新数据只发一次"，在网络带宽有限时非常高效。
4. **双端口地图传输**：普通数据走 8989 端口的 TLV 协议；多层压缩地图走独立的 8888 端口，使用块差分压缩协议，降低地图传输带宽。
5. **非阻塞与超时设计**：TCP 读写均配合 select 超时，保证主循环不会因为网络阻塞而卡住，同时能及时检测到客户端离线。
