# slam 具体代码逻辑

> 本文档描述 slam（建图进程）的内部代码逻辑，包括启动流程、多线程模型、主建图循环、SLAM 状态机、命令处理、地图后处理、虚拟墙管理、RPC 交互与轨迹记录，不含源码。

---

## 1. 启动流程

### 1.1 信号注册
注册中断与终止信号处理器，用于优雅退出。

### 1.2 实例化核心类
创建 SLAM 主类对象，其构造函数完成各类滤波器、状态标志、数据容器的默认初始化。

### 1.3 启动服务
调用 Start 方法，依次完成：
- 设置 LCM 消息订阅回调，绑定激光、里程计、SLAM 命令、AMCL 交互等多个通道。
- 创建并绑定主建图线程，运行主循环函数。
- 启动 RPC 服务，对外提供房间编辑、重定位等接口。

### 1.4 主进程守活
入口函数进入循环，每 200 毫秒检测一次退出标志，直到收到终止信号后调用 Stop 进行线程回收。

---

## 2. 多线程模型与职责划分

系统内部主要存在三条工作线程，分工明确：

### 2.1 主建图线程（mThread）
- **职责**：运行主循环方法，是 SLAM 的核心计算线程。
- **工作内容包括**：参数加载、历史地图加载、主建图循环、GMapping 调用、地图更新与发布、地图后处理、房间分割、状态机切换。
- **触发特点**：以条件驱动的方式运行，当满足建图条件时执行建图更新，否则按状态进入休眠或周期性发布地图信息。

### 2.2 消息与事件处理线程（sThread）
- **职责**：运行事件处理方法，处理异步事件与数据同步。
- **工作内容包括**：调用 LCM 消息处理消费消息队列；周期性地执行激光与里程计数据按时间戳对齐后推入同步队列；在已建图状态下周期性调用新房间探测；在需要时触发增量式地图更新。
- **时序控制**：该线程以 20 毫秒为周期运行，若超时则跳过睡眠以保证实时性。

### 2.3 RPC 服务线程
- **职责**：运行独立的 REST-RPC 服务器，监听外部客户端请求。
- **工作内容包括**：接收 APP 下发的房间编辑指令、重定位请求、地图查询等，将原始命令解析后暂存，供主建图线程在下一轮循环中读取并执行。

**数据交互方式**：各线程之间通过互斥锁保护共享数据。激光原始数据、里程计数据、同步后的数据队列、地图数据、房间多边形数据等均使用独立的锁进行并发控制。

## 3. 主建图循环逻辑

主建图循环位于主线程内，其逻辑可划分为数据同步、建图触发、GMapping 调用、地图更新四个阶段。

### 3.1 激光与里程计数据处理
- 激光数据通过 LCM 回调进入原始队列，里程计数据同理。
- 消息处理线程会周期性地从原始队列中取出最新数据，利用时间戳进行插值查找，只有当激光与对应里程计的时间误差小于 20 毫秒时，才将配对数据推入同步队列。
- 系统还维护了碰撞事件的时间戳队列，若某帧激光时间落在碰撞后的保护窗口内，则丢弃该帧以避免脏数据。

### 3.2 时间同步与数据对齐
- 使用环形缓冲区形式的时间滤波器分别缓存激光、里程计、IMU 数据。
- 在建图线程需要数据时，以激光时间戳为基准，去里程计队列中查找最近的一帧，并计算时间差。若差值过大则放弃本次更新。
- 对于 AMCL 提供的位姿，同样通过时间查找机制与内部数据对齐。

### 3.3 GMapping 调用流程
- **初始化**：收到第一帧有效激光后，调用初始化流程。根据激光参数创建测距传感器模型，配置 GMapping 的运动模型、重采样阈值、粒子数、地图边界等参数，初始化粒子滤波器。
- **扫描处理**：每次主循环触发时，将同步后的激光扫描与对应的激光位姿（由里程计转换得到）封装为测距读数，传入 GMapping 处理函数。
- **位姿输出**：处理完成后，从最优粒子中提取位姿作为 SLAM 位姿，并发布到 LCM 通道供其他模块使用。
- **地图对齐补偿**：在初始旋转阶段，系统会累积旋转角度。当旋转超过阈值后触发地图对齐流程，计算最优旋转角，并修正所有历史粒子的位姿以及充电站坐标。

### 3.4 地图更新与发布触发条件
地图更新在以下任一条件满足时触发：
- 首次建图（尚未获得地图）。
- 距离上次地图更新已超过配置的时间间隔。

更新时，系统从最优粒子的轨迹树中重建占用栅格地图。若地图尺寸发生变化（如机器人探索到新区域导致边界扩展），则重新分配地图内存。更新完成后，地图会通过 LCM 发布到 APP、AMCL、Monitor 等不同订阅方。

## 4. SLAM 状态机

系统通过一组关键布尔标志的组合来表达状态流转：

| 核心标志 | 含义 |
|---------|------|
| 已存在地图 | 是否已存在有效地图（从文件加载或建图完成） |
| 允许建图 | 是否允许进行 SLAM 建图计算 |
| 已收首帧 | 是否已收到第一帧激光并完成初始化 |
| 开始对齐 | 是否已累积足够旋转角度，可以开始地图对齐 |
| 对齐完成 | 地图对齐是否已完成 |
| 开始输出 | 是否已正式进入建图输出阶段 |
| 新房间更新 | 是否正在执行新房间增量更新 |
| 分割完成 | 房间分割是否已完成 |
| 已保存 | 地图是否已被保存到文件 |
| 重定位状态 | 重定位状态（需要发送初始位姿 / 已发送） |

### 4.1 典型状态流转
1. **Idle（空闲/待机）**：进程启动后，尝试加载历史地图。若加载成功，则进入待机状态，定期发布地图信息。
2. **Mapping（建图）**：收到开始建图命令后，系统接收传感器数据，初始化 GMapping，逐步构建地图。当累积旋转角度足够时，触发对齐，完成后正式进入建图输出。
3. **Saving（保存）**：收到保存命令后，执行保存流程。包括清除探索轨迹上的障碍物噪点、重新执行地图后处理与房间分割、保存地图文件、发布地图到 AMCL 与 APP。
4. **Loading（加载）**：启动时若检测到地图文件，加载原始栅格地图，并立即触发一次完整的地图后处理与房间分割流程。
5. **Relocalizing（重定位）**：建图初期或位姿丢失时，SLAM 会向 AMCL 发布初始位姿命令，等待 AMCL 反馈定位结果。

## 5. SlamCmd 命令处理流程

LCM 通道上的建图命令是控制 SLAM 行为的主要外部接口。

### 5.1 开始建图
- 若当前无地图：直接开启 SLAM 计算。
- 若已有地图：支持新地图模式，可在已有地图基础上继续探索或更新。

### 5.2 停止建图
将允许建图标志置为 false，停止调用 GMapping。记录当前里程计与 AMCL 位姿，用于后续可能的位姿补偿。若处于新房间更新模式，则保存当前增量地图并触发重定位。

### 5.3 重置建图
- 若此前已保存地图但当前无地图：从文件恢复地图并重新执行房间分割。
- 若已有地图且正在建图：根据当前 AMCL 位姿重新计算房间信息、更新充电站区块。
- 若首次建图：重置内部状态，清空历史数据，向 AMCL 发送特殊命令以配合全新建图。

### 5.4 保存地图
由外部任务完成状态消息触发，设置保存命令标志，在主循环中异步执行保存与后处理。

### 5.5 删除/重置地图
删除地图文件，清空内存地图，重置所有状态标志，并通知 AMCL 进行相应复位。

此外，系统还监听来自 AMCL 的反向命令通道，处理重定位成功/失败、里程计重置等事件。

---

## 6. 地图后处理流程

地图后处理发生在两种场景：加载历史地图时、保存地图时。

### 6.1 原始地图提取
将内存中的栅格地图数据转换为图像格式。

### 6.2 OpenCV 图像优化
调用地图优化函数对地图进行处理，包括：
- 膨胀与腐蚀操作，用于消除细小噪点和平滑边界。
- 滤波处理，分离墙壁与自由空间。
- 生成用于显示和用于分割的两幅不同地图。

### 6.3 房间分割
调用房间分割函数，其内部综合使用以下算法：
- **Voronoi 图**：提取骨架，识别房间之间的通道。
- **形态学处理**：进一步净化地图结构。
- **波前/区域生长**：从机器人当前位置开始，将自由空间划分为不同房间区域。

### 6.4 索引图生成
得到每个像素所属的房间 ID 图。

### 6.5 房间多边形生成
将分割后的房间轮廓转换为多边形，计算每个房间的入口点、朝向以及相邻房间连接关系。

### 6.6 地图数据编码
将房间 ID 编码到地图字节的高 3 位中（低 3 位保留障碍物/自由/未知信息），生成供 APP 显示的优化地图。

### 6.7 结果发布
通过 LCM 发布房间多边形数组和优化后的地图。

## 7. 虚拟墙与禁区管理

虚拟墙与禁区是房间编辑的核心功能，其管理逻辑贯穿 RPC 接收、数据转换、房间归属判定、持久化存储和形态学再处理。

### 7.1 数据解析与存储
- 外部通过 RPC 以特定 DPID 下发虚拟墙（线段）和禁区（多边形）的原始坐标。
- RPC 层将原始字节流解析为世界坐标系下的浮点坐标序列，暂存在房间编辑结构中。
- 主线程在循环中检测到存在待处理的房间编辑命令时，调用编辑函数。

### 7.2 归属判定与分配
- **虚拟墙**：将线段坐标转换为地图像素坐标后，沿线段采样并查询索引图，获取线段穿过的所有房间 ID。随后以该线段为中心生成一个窄多边形，将该多边形顶点坐标写入对应房间的虚拟墙列表。
- **禁区**：将多边形顶点转换到像素坐标后，使用图像绘制填充掩码。遍历掩码覆盖区域内的每个像素，统计各房间 ID 的像素数量，将面积占比高的房间判定为归属房间，并将该多边形顶点坐标写入对应房间的禁区列表。

### 7.3 持久化与再处理
- 所有编辑结果序列化保存到二进制文件，下次启动时可通过读取函数恢复。
- 当虚拟墙或禁区发生变更后，系统调用更新函数：将所有房间的禁区绘制到一张掩码图上，进行形态学膨胀处理，然后重新计算受影响的房间多边形边界，确保导航与清洁任务使用的房间几何是准确的。

## 8. SlamRpc 接口逻辑

SLAM 的 RPC 服务器为外部模块（APP、HMI、Localization）提供以下能力：

### 8.1 APP 指令通道
- 接收标准命令和原始字节命令。
- 原始命令覆盖：虚拟墙设置、禁区设置、房间分割、房间合并、房间命名、清扫顺序调整、房间属性（吸力、水量等）设置、地图重置、地图保存。

### 8.2 Localization 交互
- 供定位模块查询地图状态。
- 向 SLAM 下达重定位控制指令（开始重定位、开启/关闭重定位检测）。

### 8.3 HMI 交互
- 处理屏幕端发起的重定位请求。
- 处理 HMI 下发的控制命令。

### 8.4 机器人状态更新
接收上层广播的机器人状态机与工作模式。

### 8.5 坐标转换
RPC 层维护了一个基于地图原点与分辨率的简易坐标转换函数，用于将 APP 下发的像素级坐标转换为 SLAM 世界坐标（米）。

## 9. 与 Localization（AMCL）的交互逻辑

SLAM 与定位模块之间通过多条 LCM 通道进行双向协同：

### 9.1 SLAM 订阅的 AMCL 通道
- **AmclOdom**：接收 AMCL 输出的机器人位姿及定位置信度分数。SLAM 利用该分数监控定位质量：若连续多帧分数低于阈值，标记为异常；若连续多帧高于阈值，则恢复。在特定模式下，SLAM 还会使用该位姿对 GMapping 的粒子分布进行补偿修正。
- **AmclPoseInfo**：接收 AMCL 位姿详情，用于新房间探测时的位姿质量评估。
- **AmclToSlamCmd**：接收 AMCL 的控制反馈，如重定位成功、重定位失败、里程计已重置等。

### 9.2 SLAM 发布的 AMCL 通道
- **AmclMap / MAP**：向 AMCL 发布当前栅格地图。保存地图后，会显式发布一次完整地图供 AMCL 更新。
- **GmOdom**：向外部发布 SLAM 计算出的最优位姿。
- **InitAmclPose**：在建图初期或位姿丢失时，向 AMCL 发送初始位姿（高斯分布，小协方差），触发 AMCL 重定位。
- **SlamToAmclCmd**：发送控制命令字，例如通知 AMCL 开始配合全新建图、通知 AMCL 地图已保存可切换地图源、开启或关闭重定位检测等。

### 9.3 协同流程
1. 新地图构建开始时，SLAM 若检测到需要重定位，会发布初始位姿。
2. AMCL 定位成功后反馈，SLAM 继续建图。
3. 建图过程中，SLAM 持续监听 AMCL 分数，若定位质量差，可标记异常位姿帧。
4. 保存地图后，SLAM 发布完整地图和命令，AMCL 据此加载新地图并继续定位。

## 10. 轨迹记录与 AppTraj 生成逻辑

系统中存在多条轨迹数据流，服务于不同的业务目的：

### 10.1 传感器轨迹记录（调试/回放用途）
当配置使能传感器保存标志时：
- 每次收到激光消息，将原始激光扫描数据追加到传感器轨迹向量。
- 每次收到里程计消息，将里程计位姿追加到里程计轨迹向量。
- 保存地图时，若该标志为真，则调用序列化函数将上述两条轨迹写入二进制文件，可用于离线回放与问题分析。

### 10.2 新房间探测轨迹
在已建图且处于清洁状态时，事件处理线程周期性地调用新房间探测函数：
- 该函数将当前 AMCL 位姿持续追加到探测轨迹向量中，形成一条随时间延伸的机器人轨迹。
- 当系统判断机器人可能进入新区域（检测到足够大的未知空间且已知自由空间占比小）时，会记录一个入口位姿。
- 后续处理中，系统会分析探测轨迹与入口位姿的对应关系，从轨迹中提取一段作为虚拟墙的基础数据，用于构建新房间的边界。

### 10.3 初始建图轨迹
在开始建图但尚未完成地图对齐的阶段，系统会将原始里程计和激光数据分别缓存到初始向量中。地图对齐完成后，这些数据用于辅助计算对齐参数，随后被清空。
